# 点餐系统需求文档

## 1. 项目背景

当前已有系统为一个基于 uni-app 的点餐小程序前端原型，已具备基础的点餐、购物车、订单查看、管理员菜品管理等演示能力。

为了形成完整的业务闭环，后续建议扩展为以下三端协同架构：

1. **小程序端**：面向顾客的点餐与订单查询系统
2. **后台管理系统**：面向店长/管理员的 Web 管理平台
3. **Spring Boot 3 后端系统**：提供统一业务接口、权限控制、数据存储与管理能力

---

## 2. 建设目标

构建一个完整的餐饮点餐系统，实现以下目标：

- 顾客可在小程序完成菜品浏览、加入购物车、提交订单、查看订单进度
- 管理员/店长既可登录后台管理系统，也可登录小程序处理核心业务
- 管理员可在后台管理菜品、分类、订单、用户、门店信息
- 后端系统提供统一 API、权限认证、订单流转、数据持久化和统计能力
- 三端数据统一，解决当前前端原型中本地模拟数据不一致的问题

---

## 3. 系统组成

## 3.1 小程序端
面向顾客以及管理员/店长，既支持前台点餐业务，也支持轻量化的门店运营操作。

## 3.2 后台管理系统
面向管理员、店长、运营人员，主要负责业务配置与运营管理。

## 3.3 Spring Boot 3 后端系统
面向系统支撑，主要负责业务逻辑处理、数据库访问、权限校验、接口输出。

---

## 4. 角色设计

建议系统至少包含以下角色：

| 角色 | 说明 |
|---|---|
| 顾客 | 使用小程序浏览菜品、下单、查看订单 |
| 店员/接单员 | 可在后台或小程序查看订单、接单、更新进度、确认上菜 |
| 店长/管理员 | 可在后台或小程序管理菜品、分类、订单、门店配置 |
| 超级管理员（可选） | 负责系统级配置、账号权限分配 |

---

## 5. 小程序端需求

## 5.1 首页/菜单页
### 功能说明
用户进入小程序后，可查看菜品分类和菜品列表。

### 可能功能
- 菜品分类展示
- 菜品列表展示
- 菜品图片、名称、价格、销量展示
- 菜品搜索
- 热门推荐/招牌菜展示
- 菜品详情跳转

### 需求要点
- 支持从后端动态加载分类和菜品数据
- 支持按分类筛选菜品
- 支持展示菜品状态，如“上架/售罄”
- 支持分页或下拉刷新

---

## 5.2 菜品详情页
### 功能说明
展示单个菜品的详细信息。

### 可能功能
- 多图轮播
- 菜品名称、价格、月售、描述
- 规格选择（可选）
- 口味选择（可选）
- 加入购物车
- 立即下单

### 需求要点
- 根据菜品 ID 查询真实数据
- 展示库存/售卖状态
- 支持不同规格价格展示

---

## 5.3 购物车
### 功能说明
用户选择菜品后进入购物车结算。

### 可能功能
- 添加商品
- 减少商品数量
- 删除商品
- 清空购物车
- 显示总件数与总价
- 备注信息填写

### 需求要点
- 购物车数据应与登录用户绑定
- 可本地缓存，也可服务端存储
- 下单前需要进行库存与菜品状态校验

---

## 5.4 下单
### 功能说明
用户确认菜品后提交订单。

### 可能功能
- 提交订单
- 选择就餐方式（堂食/外带）
- 选择桌号（堂食场景，可选）
- 填写联系人/手机号（外带场景，可选）
- 填写备注
- 微信支付（后续接入，可选）

### 需求要点
- 生成唯一订单编号
- 记录订单明细、金额、下单时间
- 初始订单状态为“待接单”
- 下单成功后清空购物车

---

## 5.5 我的订单
### 功能说明
用户查看自己历史订单与订单状态。

### 可能功能
- 订单列表
- 订单详情
- 查看订单状态流转
- 再来一单
- 取消订单（待接单时）
- 评价订单（可选）

### 需求要点
- 按当前登录用户过滤订单
- 支持状态分类：待接单、已接单、制作中、已完成、已取消
- 支持订单详情查看

---

## 5.6 用户中心
### 功能说明
展示当前用户信息及常用功能入口。

### 可能功能
- 用户头像昵称展示
- 登录/退出登录
- 收货地址管理（外带场景）
- 我的订单入口
- 联系商家
- 意见反馈（可选）

### 需求要点
- 用户登录信息需持久化
- 用户身份与权限应由后端统一返回

---

## 5.7 管理员/店长小程序工作台
### 功能说明
管理员或店长登录小程序后，不再只看到普通顾客页面，而是进入适配移动端的小程序工作台，用于处理门店核心业务。

### 可能功能
- 查看待处理订单
- 查看订单进度
- 接单
- 更新订单状态
- 完成上菜
- 查看当前菜单要求/菜品供应情况
- 新增菜品
- 编辑菜品
- 下架菜品

### 需求要点
- 管理员/店长使用小程序登录后，应根据角色切换到管理视图
- 工作台需突出“待接单、制作中、待上菜、已完成”等关键状态
- 订单操作应适配移动端场景，按钮简洁明确
- 菜品操作以轻量管理为主，支持快速新增、编辑、上下架
- 小程序端管理能力应受角色权限控制，避免普通顾客误访问

---

## 5.8 小程序端订单进度管理
### 功能说明
管理员/店长在小程序中处理订单流转，并让顾客实时查看订单进度。

### 可能功能
- 查看待接单订单
- 接单
- 标记制作中
- 标记待上菜/待取餐
- 标记已上菜/已完成
- 拒单或取消订单
- 查看订单详情和备注

### 建议订单状态
- 待接单
- 已接单
- 制作中
- 待上菜
- 已上菜
- 已完成
- 已取消

### 需求要点
- 管理员/店长可在小程序端直接推进订单状态
- 每个状态变更需记录操作人和时间
- 顾客端应同步显示最新订单进度
- 状态流转应符合业务规则，不能跳过关键节点

---

## 5.9 小程序端菜品管理
### 功能说明
管理员/店长在小程序端对菜品进行轻量化管理。

### 可能功能
- 新增菜品
- 编辑菜品名称、价格、分类、描述、图片
- 上架菜品
- 下架菜品
- 查看当前菜品列表
- 按分类或状态筛选菜品

### 需求要点
- 小程序端应支持基础菜品维护，便于门店现场快速调整
- 下架菜品后，顾客端菜单中不再展示或标记不可售
- 菜品新增/编辑/下架后应同步到后台管理系统和后端数据库
- 若管理员在小程序端操作菜品，需保留操作日志

## 7. Spring Boot 3 后端系统需求

## 7.1 技术定位
后端系统采用 Java + Spring Boot 3 构建，作为统一服务端。

### 建议技术栈
- Java 17+
- Spring Boot 3.x
- Spring Web
- Spring Validation
- Spring Security
- MyBatis-Plus 或 JPA
- MySQL 8.x
- Redis（可选，用于缓存、登录态、热点数据）
- MinIO/本地存储（用于图片上传，可选）
- Knife4j / springdoc-openapi（接口文档）
- JWT（后台登录鉴权）

## 8. 核心业务流程

## 8.1 用户下单流程
1. 用户进入小程序
2. 浏览分类与菜品
3. 加入购物车
4. 确认订单信息
5. 提交订单
6. 后端生成订单与订单明细
7. 管理后台收到待处理订单
8. 店员/管理员接单并更新状态
9. 用户在小程序查看订单状态

---

## 8.2 管理员处理订单流程
1. 管理员登录后台或小程序工作台
2. 查看待接单订单
3. 接单或拒单
4. 更新订单状态为制作中
5. 堂食场景下标记待上菜
6. 完成上菜后标记已上菜或已完成
7. 用户在小程序同步看到状态变化

---

## 8.3 菜品管理流程
1. 管理员进入后台管理端或小程序菜品管理页面
2. 新增或编辑菜品信息
3. 上传菜品图片
4. 设置分类、价格、状态
5. 根据实际经营需要执行上架或下架
6. 保存后同步到前台菜单页与管理端列表

---

## 8.4 管理员/店长小程序运营流程
1. 管理员/店长登录小程序
2. 进入移动端工作台
3. 查看待处理订单和当前订单进度
4. 根据现场情况处理接单、制作、上菜、完成等操作
5. 如菜单有变动，可直接新增菜品、编辑菜品或下架菜品
6. 顾客端菜单与订单状态实时同步更新

---

## 10. 非功能需求

## 10.1 性能要求
- 常规查询接口响应时间应尽量控制在 1 秒内
- 支持日常门店级别订单处理场景

## 10.2 安全要求
- 后台接口必须鉴权
- 管理员密码加密存储
- 关键操作需记录日志
- 接口参数必须校验

## 10.3 可维护性要求
- 前后端分离
- 接口统一返回结构
- 明确分层：Controller / Service / Mapper
- 提供接口文档

## 10.4 可扩展性要求
- 后续可扩展微信支付
- 后续可扩展评价系统
- 后续可扩展优惠券、会员积分、配送管理

---

## 11. 当前原型与正式系统的差距

根据现有 uni-app 原型，当前仍存在以下差距：

1. 当前数据为本地写死，未接数据库
2. 菜品管理与前台菜单未使用统一数据源
3. 订单页面与订单数据结构不完全一致
4. 登录/权限为演示逻辑，未形成真实账号体系
5. 购物车与订单未持久化
6. 未形成后台管理 Web 端
7. 未实现 Spring Boot 后端接口与数据库设计

---
